**Stories:**
- **6.1** — Bot real-time edit/delete detection. Publishes `discord.message.updated`/`deleted` to two new streams; `Partials.Message`/`Channel` on the client (without which discord.js silently drops uncached events). Reuses the `messageCreate` handler shape (publish-only, no transaction). Two review rounds. **Real bug caught:** the bot-author guard dereferenced `author` *before* the partial `fetch()` → `TypeError` on an uncached edit (the exact case partials exist for); fix moved `fetch()` first. All 3 review layers flagged it.
- **6.2** — Worker Sync (idempotent re-index / purge). Second consumer in `@share2brain/workers`, own Redis client(s), gated by `config.sync.enabled`, PEL-replay on boot, XACK-only-after-success (AD-13). Reconciled the epic's `message_id` pseudocode against the real grouped `message_ids[]` schema. One review round, 2 decisions resolved with Borja: **one Redis client per loop** (a shared client serialized the two blocking reads) and **embed moved outside the DB transaction** (was holding row locks + a pooled connection across the embeddings HTTP call, superseding AC-1's literal "in one transaction" wording).
- **6.3** — Startup offline reconciliation. Publish-only; diffs persisted rows against a fresh Discord window and republishes via the reused 6.1 handlers. **Real boundary bug caught in integration testing:** the `last_seen` anchor is excluded from the re-fetch but is still the newest persisted row, so the `rowKey > lastSeenKey` guard would have concluded a **false delete of the newest message every boot** — tightened to `>=` + regression test. Round 2 re-reviewed the 3 patches as new code and verified empirically they fail against the pre-patch snapshot.
- **6.4** — Notifications + security + graceful shutdown. Inline `Notifier` in `@share2brain/shared` (bounded, non-throwing, no-op when disabled — deliberately NOT the reserved stream, since a crashing process can't rely on XADD); helmet (HSTS/nosniff/`DENY`/CSP, CORP tuned for the credentialed cross-origin SPA); 3-tier rate limiting injected via `AppOptions` so it defaults OFF in tests/e2e; real backend `server.close()` + 10s drain + bounded `redis.quit()`/`db.end()` + reentrancy guard. **Two real gotchas:** helmet's `frameguard` defaults to `SAMEORIGIN` (AC needed `DENY`), and `interpolateEnv` substitutes `${VAR}` even inside YAML comments (would abort `loadConfig()` on a fresh clone). One review round, 5 patches (Notifier secret-redaction + 4096-char truncate, `isShuttingDown` guard on fatal handlers, workers grace 30s→35s, `trust proxy` gated).

## 2. What Went Well

- **The "reality surfaces late" defects were caught inside the loop this time — 0 user-facing incidents.** The pattern that bit Epics 3/4/5 (4096-vs-1536 dims, Playwright-not-installed, the Anthropic multi-`system` 400 that only surfaced when Borja chatted live) was *contained*: 6.3's false-delete anchor bug surfaced in **integration testing**; 6.1's null-author-on-partial and 6.4's helmet/`interpolateEnv` gotchas surfaced in **review/implementation**. Nothing reached production. **Borja confirmed as a highlight.**
- **The Epic 5 testing lessons paid off directly.** 6.2/6.3 ran integration tests against real Postgres + Redis with idempotency (AD-13) asserted (redeliver → identical state). No fake that could lie — the exact remedy for the Epic 5 king defect.
- **"Every patch is new, un-reviewed code" (Epic 3 AI#1) still earning its cost.** 6.1 and 6.3 ran a **second review round** over the round-1 patches; 6.3 round 2 verified empirically that its regression tests fail against the reconstructed pre-patch snapshot — the Epic 5 "a test must discriminate its fix" rule, applied. **Borja confirmed the review/test discipline as a highlight.**
- **Zero migrations, zero shared-contract changes across four stories.** Every story reused existing schema columns (`deleted_at`, `message_ids[]`) and the event contracts already in `@share2brain/shared`. Clean reuse of the 3.1/3.2/3.3 handler/backfill/Indexer patterns.
- **Long-standing debt finally paid.** The **real backend graceful shutdown** — open since Epic 1 AI#4, deferred through 5 epics — landed in 6.4 with bot/workers parity, plus the previously-absent backend `uncaughtException`/`unhandledRejection` handlers. **Borja confirmed the debt-payoff as a highlight.**

## 3. Challenges

- **🟡 `MAXLEN` / stream trimming — the trigger fired and it was NOT addressed (third deferral).** The Epic 5 retro explicitly flagged that 6.1 adds two new unbounded streams, re-triggering Epic 3 AI#4. 6.1 shipped with bare `xAdd`, no `MAXLEN`. It is now the top item of the operational backlog (§5) — with no next epic, it was "now or never." **Borja flagged this class of homeless debt as his main concern.**
- **🟡 The recurring RBAC integration-test flake.** `rbac.integration.test.ts` / `channels.integration.test.ts` fail non-deterministically ("test-guild role leak" — a stale shared `channel_permissions` row). Reported as pre-existing/unrelated in **three** consecutive Completion Notes (6.2, 6.3, 6.4) and first seen in Epic 4. Never closed. It erodes trust in the integration gate ("is this failure mine or the flake?").
- **🟡 Deferred debt with no future epic to fall into.** As the final epic, everything deferred (transactional outbox, Sync DLQ/retry-cap, `messageDeleteBulk`, edit-to-anchor full-reconcile) lost its "next epic" home. Left implicit, these die silently. **Resolved in §5:** an explicit post-roadmap operational backlog.

## 9. Key Takeaways (Epic 6 and the arc of the project)

1. **The compounding lessons worked.** Epic 6's clean run — 0 user-facing incidents, real bugs caught in test/review — is the direct payoff of Epic 3's "every patch is new code," Epic 4's run-unique isolation, and Epic 5's "test against reality, and a test must discriminate its fix." The process debt paid down over five retros showed up as fewer escaped defects in the last one.
2. **"Reality surfaces late" is beatable by moving reality earlier.** Integration tests against real Postgres/Redis caught 6.3's false-delete boundary bug before a single boot in production would have — the same defect class that reached the user in Epics 3–5.
3. **Deferred debt needs an explicit home, or it dies silently.** Three deferrals of `MAXLEN` across three retros is the proof. The operational backlog exists so the roadmap's end doesn't quietly discard reliability work.
4. **Boring, reused patterns win.** Four stories, zero migrations, zero shared-contract changes — reusing the handler/backfill/Indexer shapes made the last epic the lowest-drama one.
5. **Done ≠ closed.** The product is functionally complete, but P1.1/P1.2 are real operational gaps. "Functionally complete" and "operationally closed" are different bars — the backlog names the distance between them.
