## 2. What Went Well

- **The Epic 2 early-validation spike (Action Item #1) paid off directly.** Both new external integrations — `discord.js` Gateway and the embeddings API — were validated against the real service *before* building 3.1/3.3 on top. It paid off immediately in 3.0: the real embeddings endpoint returned corrupt all-zero 1024-dim vectors instead of the configured length, and the `assertEmbeddingDimensions` guard (AC-6) caught exactly that. This broke the "external-integration reality surfaces late" pattern that bit Epic 1 (7-vs-8 tables) and Epic 2 (ioredis).
- **The cross-story intelligence loop kept closing deferred debt.** Each story resolved the previous one's residuals: 3.2 made the producer idempotent (resolving 3.1's deferred `onConflict` finding); 3.3 resolved 3.1's deferred "XADD-before-COMMIT" concern by reading `content` from the event itself and leaving the entry pending (no XACK) when the row hasn't landed. Zero re-work from inter-story misalignment.
- **Adversarial code review caught real correctness bugs, not noise.** 3.3 round 1: a `chunk_key` collision between duplicate `messageId`s across two grouping windows in the same batch that **silently overwrote** already-indexed content (silent data loss, zero error signal); plus an infinite crash-loop on tombstoned (XDEL'd) PEL entries (`message == null`). 3.2: **permanent** message loss via cursor advancement past a transiently-failed message. All caught before any deployment. **Borja confirmed: the review was a valuable safety net.**
- **AD-13 idempotency made concrete and tested against real infra.** PEL-as-DLQ (no retry-max, no per-entry quarantine), XACK only after success, idempotent UPSERT against real pgvector, dimension guard. Real-infra integration tests for Streams + pgvector — Epic 2 Action Item #2, delivered.
- **Hardening pulled forward (Epic 2 Action Item #3).** `uncaughtException`/`unhandledRejection` + reconnect-with-backoff in bot and workers; bounded shutdown via `Promise.race` timeouts on every teardown step.

## 3. Challenges

- **High review churn, and patches that introduced their own bugs.** 3.2 needed 4 independent passes; 3.1 needed 3. Several times a defect was introduced *by a prior pass's patch*: a latent build error from 3.2's 1st pass, the `BigInt('') === 0n` gap inside 3.2's own 2nd-pass fix, and 3.1's P4 "fail-fast" that was **dead code** (node-redis `reconnectStrategy` always returns a number, so `connect()` never rejects). **Resolution/insight (Borja):** this was not "bad churn" — each new independent pass caught the previous pass's defective patch, so the system self-corrected. The lesson is not "review less"; it is **treat every patch as new, un-reviewed code** and verify it with the same rigor as the original.
- **Config-vs-reality, again.** Configured dimension 4096 vs. migrated schema `vector(1536)` (pgvector HNSW cannot index >2000 dims), plus the real endpoint returning corrupt 1024-dim vectors. Resolved (1536, MRL, spike-gated), but it is the same "external-integration reality surfaces late" pattern seen with ioredis and 7-vs-8 tables.
- **Accepted residual risks that couple across stories.** 3.2's per-message retry **amplified** 3.1's COMMIT-after-XADD race (from 1 tolerated orphan event to up to 3 duplicate events), relying on 3.3's Indexer deduping by `messageId` as the safety net — which 3.3 did implement. It closed cleanly, but it was a three-story coupling of accepted risks resting on AD-13.
- **Deferred debt accumulating:** no `MAXLEN`/trimming → `share2brain:discord:messages` grows unbounded; transactional outbox (XADD-inside-tx holds a pg connection across the Redis round-trip → pool exhaustion risk); thread messages don't match a configured parent channel; Discord 429 retry; log redaction.

## 6. Next Epic Preview — Epic 4: Search, Documents & Read Tracking

**Stories:** 4.1 Backend semantic-search API (query embed + pgvector search with **RBAC inside the query**, AD-12) · 4.2 Backend documents + read-tracking API · 4.3 Web Search view · 4.4 Web Documents view + read-tracking UI + sidebar badge.

**Dependencies on Epic 3 — all present and verified:** the `embeddings` table populated by the Indexer, `chunk_key`, and the embeddings provider factory (to embed the query). RBAC `findAllowedChannelIds` (from Epic 2, AD-12) is reused as-is.

**Significant discoveries (design notes for Story 4.1/4.3 — Borja's decision: NOT a dedicated design session, resolve at story creation):**
1. **`deleted_at` exclusion over grouped chunks** — `deleted_at` lives on `discord_messages`, not `embeddings`, and a chunk groups multiple messages (`message_ids` array). Excluding a chunk whose message was deleted requires a JOIN and a semantics decision (any vs all). `messageDelete` is Epic 6, so `deleted_at` is always NULL today — but the 4.1 query must be written correctly from the start. → Action Item #2.
2. **Read tracking over grouped chunks** — read status (4.2) is per `embeddingId` (the grouped chunk), which is consistent; but the "view in Discord" link (4.3) must pick which `messageId` of the group to link. → Action Item #3.

**No re-plan required for Epic 4.** No Epic 3 assumption invalidates the Epic 4 plan; both discoveries are query-design decisions to nail at Story 4.1/4.3 creation, handled by the cross-story intelligence loop.

## 9. Key Takeaways

1. Early-validation spikes work: validating external integrations before building on them broke the "reality surfaces late" pattern that cost Epic 1 and Epic 2 real re-work.
2. A code-review patch is new, un-reviewed code — verify it with the same rigor as the original. The value came precisely from independent passes catching prior passes' defective patches.
3. The cross-story intelligence loop + real-infra integration tests are the two process assets that produce a trustworthy "done" — extend them to Epic 4's search/RBAC queries.
4. Accepted residual risks can couple across stories (3.1→3.2→3.3 orphan-event chain). When you accept a residual risk, note which downstream invariant (here AD-13 idempotency) is the safety net, so a later story doesn't unknowingly break it.
