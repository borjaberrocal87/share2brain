- (constraint) PRD Final (docs/PRD.md) — decisiones SD-1..SD-18 heredadas como invariantes de partida: Discord como fuente, YAML config, Docker Compose, LangChain.js, pgvector, Redis Streams, Read Tracking per-embedding, RBAC por canal, single-tenant
- (decision) AD-1: Topología de runtime — Bot, Backend y Workers como tres procesos Node.js independientes, cada uno en su propio servicio Docker Compose. Binds: cada servicio tiene su propio Dockerfile y ciclo de vida. Prevents: que un backfill pesado degrade la API REST; que un crash del Bot derribe el agente.
- (decision) AD-2: Monorepo con npm workspaces — packages/bot, packages/backend, packages/workers, packages/web, packages/shared. Cada servicio tiene su propio Dockerfile. Binds: un solo repo que el operador clona; código compartido (tipos DB, clientes Redis, utils) en packages/shared. Prevents: duplicación de schemas de DB; divergencia de tipos entre servicios.
- (decision) AD-3: Web App — Vite + React SPA. Build genera estáticos servidos por nginx o el Backend. Sin servidor Node adicional en Compose. Binds: packages/web es un paquete de estáticos, no un servicio runtime. Prevents: añadir un cuarto proceso a Compose; complejidad SSR innecesaria para una app con login obligatorio.
- (decision) AD-4: Streaming del chat — SSE (Server-Sent Events) en lugar de WebSocket. Cliente hace POST /api/chat, recibe stream via EventSource. Binds: el endpoint de chat devuelve text/event-stream; no hay upgrade de protocolo. Prevents: complejidad de autenticación WebSocket; configuración extra de nginx; reconexión manual en cliente.
- (decision) AD-5: Capa de acceso a DB — Drizzle ORM. Schema definido en TypeScript en packages/shared/src/db/schema.ts. Migraciones como SQL explícito (drizzle-kit). Soporte nativo pgvector. Sin binario externo en Docker images. Binds: todos los servicios (bot, backend, workers) importan el schema y el cliente desde packages/shared. Prevents: definiciones de tabla duplicadas o divergentes entre servicios.
- (decision) AD-6: Contrato API frontend-backend — Zod schemas en packages/shared/src/schemas/. Backend los usa para validación runtime (Zod.parse). Frontend infiere tipos con z.infer<>. Binds: ningún servicio define shapes de API localmente; todo parte del schema compartido. Prevents: divergencia silenciosa entre request/response shapes del frontend y backend.
- (decision) AD-7: Entrada HTTP — nginx como servicio adicional en Compose. Sirve estáticos de packages/web/dist/ y hace reverse proxy de /api/* al Backend. Único puerto expuesto al operador (80/443). Punto natural de terminación TLS. Binds: la SPA y la API comparten origen desde el cliente; el Backend no sabe nada del frontend. Prevents: que el Backend sirva estáticos; competencia de recursos entre SPA y API requests.
- (decision) AD-8: Config loading — loadConfig() centralizado en packages/shared/src/config/. Parser Zod del Share2Brain.config.yml. Todos los servicios lo importan en main.ts; si el YAML es inválido el servicio muere en startup con error claro. Binds: ningún servicio parsea el YAML localmente. Prevents: interpretaciones divergentes del mismo campo de configuración entre servicios.
- (decision) AD-9: Migraciones DB — servicio migrator en Compose (contenedor de vida corta) que corre drizzle-kit migrate antes que los demás servicios vía depends_on condition: service_completed_successfully. El operador nunca corre migraciones manualmente. Binds: docker-compose.yml tiene 7 entradas: migrator + nginx + bot + backend + workers + postgres + redis. Prevents: estado de DB inconsistente en upgrades.
- (decision) AD-10: Sesiones de autenticación — express-session con connect-redis como store primario (Redis ya en el stack). Cookie httpOnly contiene solo el session ID. Lookups O(1) desde Redis sin query a PostgreSQL por request. La tabla sessions del PRD se elimina del schema (Redis es la fuente de verdad de sesiones). Revocación inmediata posible borrando la key. Binds: el Backend usa connect-redis; sesiones no se persisten en PostgreSQL. Prevents: un DB lookup por cada request autenticado.
- (decision) AD-11: Agent Runtime — LangChain.js v0.3+ con LangGraph. El agente RAG se modela como StateGraph (nodos: retrieve → reason → respond, con tool_call loop opcional). El historial de conversación es estado explícito del grafo, comprimido cuando supera el presupuesto de tokens. Binds: packages/backend implementa el agente con StateGraph; no usa ConversationSummaryBufferMemory ni LLMChain legacy. Prevents: divergencia entre el modelo de memoria del PRD (legacy) y el framework actual; magia oculta en el historial de conversación.
- (assumption) [ASSUMPTION] TypeScript en todos los paquetes del monorepo (bot, backend, workers, web, shared). Implícito por LangChain.js v0.3+ y Drizzle ORM.
- (decision) AD-12: RBAC aplicado a nivel de query — el vector search filtra por channel_ids permitidos para el usuario autenticado (WHERE channel_id = ANY(:allowed_channels)) antes de calcular similitud coseno. La tabla channel_permissions se materializa desde Share2Brain.config.yml en el arranque del Backend (upsert). Binds: ningún endpoint de búsqueda o chat devuelve fragmentos de canales no permitidos. Prevents: fuga de información vía resultados RAG que ignoren permisos.
