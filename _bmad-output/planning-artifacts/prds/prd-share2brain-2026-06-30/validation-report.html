Share2Brain Self-Hosted — Validation Report

docs/PRD.md
Poor

Este PRD es técnicamente sólido a nivel de infraestructura — un ingeniero podría comenzar la implementación — pero falla como documento de decisión de producto: no tiene preguntas abiertas, no hay etiquetas de asunción, no hay marcado MVP/v1 por FR, y la persona principal (Miembro) carece de journeys documentados. Un PM no puede delimitar el MVP ni validar la tesis solo con este documento.

El revisor adversarial añade tres riesgos estructurales que elevan la calificación a Poor: las métricas de éxito son aspiracionales sin base de validación, el read tracking generará una explosión de datos a escala de comunidad no trivial, y el alcance del "MVP" es en realidad un producto completo de varios meses. La falta de comportamiento definido ante caídas del API del LLM es la omisión operacional más grave.

1. Decision-readiness

Thin

La tabla de Decisiones (§14, SD-1 a SD-15) es genuinamente útil y la lista de No-objetivos (§2.3) hace trabajo real. Sin embargo, el PRD lee como si ya hubiera resuelto todos los trade-offs en una dirección, sin alternativas surfaceadas ni reconocimiento honesto de costos.

No hay preguntas abiertas en ninguna parte del documento. Para una v1.0 de un producto AI self-hosted, esto es implausible. El roadmap (§15) separa MVP de v1 a nivel de fase, pero ningún FR o componente individual está etiquetado con su fase — SB-16, SB-17, SNF-13–17 aparecen como peers de primer nivel junto a componentes MVP aunque §15 los lista como v1.

High

No hay preguntas abiertas documentadas

§ todo el documento
Un producto AI self-hosted con cero preguntas abiertas es una señal de alerta. Comportamiento de recuperación ante errores, ruta de migración de datos, modo degradado ante caída del LLM — ninguno está surfaceado para decisión del PM.
Fix: Añadir una tabla "Preguntas Abiertas" antes de §14 con al menos 5–8 ítems genuinos sin resolver, etiquetados por propietario y condición de revisión.
High

§4, §11, §15
§15 lista el alcance MVP en prosa, pero SB-16, SB-17, SNF-13–17 (sync y RBAC — listados como v1 en §15) aparecen sin etiqueta de fase en §4 y §11, creando un límite de implementación incoherente.
Fix: Añadir columna "Fase" a cada tabla de componentes y NFRs, o sufijo "(v1)" en cada fila fuera del alcance MVP.
Medium

§4.2 Agent Runtime
El happy path del agent runtime está documentado, pero no se define qué devuelve el sistema cuando el LLM está caído, cuando pgvector retorna 0 resultados, o cuando Discord API rate-limita durante el backfill.

3. Strategic coherence

Adequate

La tesis es clara: el conocimiento de comunidades Discord está fragmentado; AI self-hosted permite capturarlo y servirlo manteniendo los datos bajo control. Los objetivos SO (§2.2) mapean a esta tesis y los componentes (§4) los implementan. El ángulo de soberanía de datos (SO-7) es el diferenciador principal del producto frente a alternativas SaaS.

Sin embargo, SO-7 "Datos bajo control" está contradicho por el uso del API del LLM: el contenido de mensajes de miembros sale del servidor del operador hacia Anthropic/OpenAI para embeddings e inferencia, y esto no se reconoce nunca como tensión con SO-7. SM-3/SM-4 duplican exactamente SNF-1/SNF-2 — son targets de ingeniería reempaquetados como métricas de éxito. No hay counter-metrics nombradas en ninguna parte.

High

SO-7 "Datos bajo control" contradicho por uso del API LLM

§2.2, §4.2, §13
El sistema envía contenido de mensajes de miembros a APIs de Anthropic/OpenAI para embeddings (text-embedding-3-small) e inferencia. Nunca se reconoce como tensión con la promesa de self-hosted.
Fix: Añadir nota a SO-7 aclarando que "bajo control" se refiere a datos en reposo, y reconocer el procesamiento del proveedor LLM en §13 Threat Model.
Medium

SM-3/SM-4 duplican targets NFR y no son métricas de producto

§16
SM-3 = SNF-1 (búsqueda vectorial P95 <200ms); SM-4 = SNF-2 (agente P95 <5s). Las métricas de éxito deben medir outcomes de producto, no compliance de ingeniería.
Fix: Reemplazar SM-3/SM-4 con métricas de outcome como "% de sesiones de miembros que resultan en una respuesta encontrada."
Medium

Sin counter-metrics nombradas

§16
El modo de fallo de SM-1 (respuestas malas, alucinaciones) y SM-2 (citas que apuntan a contenido borrado/incorrecto) no tienen señales de fallo nombradas. Sin counter-metrics no hay señal de fallo en producción.
Fix: Añadir columna de counter-metric o nota a SM-1 y SM-2 nombrando el indicador de fallo (ej., "% de respuestas seguidas por la misma pregunta en 5 min").
Low

SM-5 es métrica de ingeniería en tabla de éxito de producto

§16
Cobertura de tests >80% pertenece a §11.3 como SNF-10, no a la tabla de métricas de éxito del producto.
Fix: Mover SM-5 exclusivamente a §11.3; eliminar de §16.

5. Scope honesty

Thin

La sección §2.3 de No-objetivos existe y hace trabajo real — nombra seis exclusiones específicas. Esto es positivo. Sin embargo, no hay etiquetas [ASSUMPTION] en ninguna parte del documento. Al menos cuatro asunciones son load-bearing: que el operador tiene un dominio público para los redirect URIs de Discord OAuth2; que los costos de embedding son negligibles a las escalas de comunidad objetivo; que el rate limit de Discord API es suficiente para el backfill en `backfill_limit: 1000`; que `text-embedding-3-small` a 1536 dimensiones es adecuado para la calidad de búsqueda semántica a escala de comunidad.

No hay callouts [NOTE FOR PM]. El roadmap (§15) difiere decisiones a fases sin marcar cuáles requieren una decisión PM antes de comenzar la implementación.

6. Downstream usability

Adequate

La estructura del repo (§6.3) es suficientemente detallada para que un arquitecto comience el scaffolding. Los endpoints API (§4.2) son concretos para que un frontend los stub. Los esquemas SQL (§5.2) son listos para implementación. Los archivos Docker Compose (§7) son deployables tal como están.

Sin embargo, no hay Glosario. "documento," "fragmento," "embedding," y "chunk" se usan con cierta intercambiabilidad para el mismo concepto de unidad de conocimiento indexado. "Embedding" aparece como identificador de cara al usuario en la API de read tracking (/api/read-status/:embeddingId) — expone un detalle de implementación como identificador de usuario. Los IDs de componentes (SB-1 a SB-17) cruzan tres componentes distintos en un único namespace sin mapeo visible.

High

"Embedding" expuesto como identificador de cara al usuario en la API

§4.2, §4.8
/api/read-status/:embeddingId y el flujo de read tracking referencian "embedding" como concepto clave para los miembros. Un embedding es un concepto de infraestructura; la entidad de cara al usuario debería ser "fragmento" o "documento indexado."
Fix: Introducir término de cara al usuario ("fragmento", "documento indexado") en el Glosario, mapearlo a la entidad DB, y renombrar la ruta API o añadir nota explicando su naturaleza interna.
Medium

Sin sección de Glosario

§ todo el documento
"documento," "fragmento," "mensaje," "embedding," "chunk" sin definición canónica. Causará drift de terminología en copy UX, respuestas API y codebase.
Fix: Añadir §0.2 Glosario con al menos: Operador, Admin, Miembro, Mensaje (Discord message), Fragmento (indexed chunk), Conversación, Guild, Embedding.
Medium

Namespace de IDs cruza múltiples componentes sin mapeo

§4.1, §4.2, §4.3
SB-1 a SB-17 mezcla Discord Bot, Backend y Web App en un único namespace. La creación de stories desde este PRD requiere que el autor conozca el mapeo de memoria.
Fix: Prefijar IDs por componente: BOT-1 a BOT-7 para Discord Bot, API-1 a API-5 para Backend, UI-1 a UI-5 para Web App.
Low

Sin user journeys con protagonistas nombrados

§3
Solo el Operador tiene un flujo (§3.1). El trabajo de UX y creación de stories para features de Miembro no puede source-extract de este PRD.
Fix: Añadir §3.2 Flujo del miembro con protagonista nombrado (ej., "Ana, moderadora técnica de una comunidad de developers") cubriendo el loop búsqueda/chat/read-tracking.

7. Shape fit

Adequate

Es un producto técnico centrado en el operador y el PRD está correctamente formado para eso: el flujo del operador (§3.1), configuración-como-código (§4.6), Docker Compose (§7), checklist de despliegue (§17) y Threat Model (§13) son artefactos apropiados para este tipo de producto.

El estiramiento de forma: la capa de interacción del miembro está casi ausente. Para un producto cuya propuesta de valor se entrega a los miembros (quienes buscan, chatean y leen), el PRD tiene aproximadamente 15 líneas sobre comportamiento de cara al miembro versus cientos de líneas de especificación de infraestructura. El estado "En revisión" es apropiado — este documento lee como borrador de primer autor que no ha sido revisado por PM, UX designer, ni segundo revisor técnico.

Medium

Comportamiento de cara al miembro vastamente sub-especificado respecto a infraestructura

§3, §4.3, §4.8
La Web App (§4.3) lista cinco componentes y cuatro vistas en una tabla sin detalle de interacción, empty states, error states, ni loading states. Para un PRD de producto, la capa de interfaz de la persona principal debería tener al menos un flujo de interacción.
Fix: Añadir §4.3.1 con el journey miembro de búsqueda-a-respuesta al nivel de interacción UI (no al nivel de evento de sistema), o referenciar un spec UX separado.
Low

Typo en §4.2 — "ENsamBLAR CONTEXTO"

§4.2
Casing mixto en header de sección — señala que el documento no ha sido revisado.
Fix: Corregir a "ENSAMBLAR CONTEXTO".
Low

Docker Compose version: '3.8' es sintaxis deprecada

§7.1, §7.2
Docker Compose v2 ignora la clave version y puede causar warnings en linters.
Fix: Eliminar la clave version: de ambos compose files.

Revisión adversarial

review-adversarial.md

El PRD es arquitectónicamente coherente para un proyecto de portfolio individual, pero conflate "bien especificado" con "bien pensado": las métricas de éxito son números inventados sin base de validación, el alcance del MVP es drásticamente mayor que un MVP real, y el read tracking es una bomba de tiempo de datos a cualquier tamaño de comunidad no trivial. El mayor riesgo es que el equipo entregue un stack Docker Compose que nadie puede realistamente desplegar y mantener sin experiencia DevOps, contradiciendo directamente la promesa de "despliegue trivial."

Critical

Métricas de éxito son aspiracionales sin base de validación

§16 Métricas de éxito
SM-1 (>70% preguntas resueltas sin intervención humana) y SM-2 (>95% respuestas con fuente citada) son presentadas como targets sin baseline previo, investigación de usuarios, ni benchmark. Los sistemas RAG sobre datos reales de comunidad routinamente fallan en encontrar chunks relevantes para preguntas ambiguas o jerga específica. SM-7 (>60% documentos marcados como leídos) asume un comportamiento de seguimiento de lectura voluntario sin precedente en la cultura Discord.
Fix: Reemplazar targets duros con proxies instrumentados medibles al lanzamiento, y etiquetar explícitamente todos los SM targets como "aspiracionales — a validar en beta."
Critical

Read tracking generará explosión de datos a escala de comunidad

§4.8, §5.2
user_read_status almacena una fila por par (usuario, embedding). Una comunidad con 500 miembros activos y 50.000 embeddings (fácilmente alcanzable con backfill y indexación continua) produce 25.000.000 filas en una sola tabla. El endpoint mark-all necesita insertar potencialmente 100.000 filas en una sola transacción. No hay límite de batch, paginación, ni upper bound especificado.
Fix: Definir el tamaño máximo de comunidad para el que está diseñada esta feature. Considerar un enfoque de bitset o modelo inverso "no-leído desde timestamp" en lugar de acknowledgment positivo por embedding.
Critical

MVP no es realmente un MVP

§15 Hoja de ruta
El alcance MVP listado incluye: Discord bot + backfiller + sync, web search UI, chat con RAG, read tracking, YAML config, notificaciones Telegram/Slack, empaquetado Docker Compose, y "tests básicos." Es un esfuerzo de ingeniería de varios meses para un developer individual. Más críticamente, RBAC y sync de ediciones/borrados están listados como v1 en §15, pero el spec del sistema (SB-16, SB-17, SS-9) los especifica como componentes requeridos — creando contradicción en cada PR.
Fix: Definir MVP como el artefacto deployable más pequeño donde el valor es demostrable: bot indexa + chat con citas. Mover read tracking y RBAC a v1. Hacer el límite MVP/v1 explícito en cada sección que lo referencia.
Critical

Caída del API LLM no manejada

§4.2, §11.2
El agent runtime llama un API LLM externo. Si ese API cae, el endpoint de chat presumiblemente falla. El PRD especifica graceful shutdown (SNF-8) y restart on failure (SNF-7), pero ninguno cubre el escenario de un pod saludable con LLM upstream no disponible. No hay circuit breaker definido, ni fallback en modo degradado, ni contrato de error de cara al usuario.
Fix: Definir comportamiento de fallo explícito para no-disponibilidad del LLM: al mínimo, retornar error visible al usuario con guía de retry, surfacear estado del API LLM en /health, y documentar el comportamiento en modo degradado.
High

Caída del Discord Gateway rompe la indexación en tiempo real silenciosamente

§4.1, §13
Si el bot se desconecta, los mensajes se pierden silenciosamente. El "Sync on start" solo funciona si el operador nota que el bot está caído y lo reinicia. Si el bot está caído 48 horas en un guild activo, el gap de backfill puede superar backfill_limit, creando puntos ciegos permanentes.
Fix: Especificar el algoritmo de reconciliación: almacenar el último snowflake de mensaje Discord visto por canal, y hacer backfill desde ese punto en lugar de un conteo fijo. Definir una alerta de monitoreo cuando el bot no haya recibido ningún evento en N minutos.
High

Staleness de cache RBAC crea ventana de seguridad

§4.6, §9 SS-9, SD-14
Si un admin de Discord revoca un rol a un usuario (ej., degradando un mod activo), ese usuario retiene acceso a canales restringidos hasta 5 minutos. En un escenario disciplinario esto es un problema real. El Threat Model lista esta amenaza como mitigada, pero la mitigación es incompleta.
Fix: Añadir endpoint de invalidación de cache (solo admin) para forzar refresh de roles de un usuario. Documentar el trade-off de seguridad explícitamente: "TTL por debajo de 60s causará presión de rate limit en Discord API."
High

Sin estrategia de backup o recuperación especificada

§12 Despliegue, §17 Checklist
El checklist menciona "Backup de base de datos programado" como checkbox post-despliegue, pero el PRD no especifica: frecuencia de backup, política de retención, procedimiento de restore, o qué ocurre con user_read_status después de un restore. Para un producto self-hosted, el operador es responsable — y el PRD no da orientación.
Fix: Añadir sección definiendo el contrato mínimo de backup: pg_dump diario, cadencia de snapshot de volumen, y procedimiento de restore documentado. Hacer la configuración de backup parte de Share2Brain.config.yml.
High

Historial de conversación "summary memory" sin especificar

§4.2 Agent Runtime
El agente usa "Historial de conversación (summary memory)" pero el PRD nunca define: qué estrategia de resumen se usa, cuántos turnos previos se guardan antes de resumir, cuál es el presupuesto máximo de context window, ni cómo se almacena el resumen. Es el mecanismo más probable de producir UX degradado.
Fix: Especificar la estrategia de memoria concretamente: rolling window de últimos N mensajes, o ConversationSummaryBufferMemory con presupuestos de tokens explícitos. Vincular al límite de context window del LLM configurado.
High

Tool search_web con permission: auto es una responsabilidad

§4.6 config example
La config de ejemplo habilita search_web con permission: "auto", permitiendo al agente invocar búsqueda web sin confirmación del usuario. No está en los No-objetivos, introduce dependencias externas (proveedor de búsqueda no nombrado), costo impredecible, y riesgo de que el agente cite fuentes externas que contradicen el conocimiento de la comunidad.
Fix: Mover search_web a la fase "Posterior" del roadmap. Si se mantiene, nombrar el proveedor, definir controles de costo, y cambiar el default a permission: "ask".
Medium

Estrategia de chunking arbitraria, sin optimización para Discord

§4.6 knowledge config
chunk_size: 500 es el default de LangChain, no una elección optimizada para Discord. Los mensajes cortos conversacionales (mayoría del contenido Discord) serán sobre-fragmentados. La interacción entre grouping_window: 10 y chunk_size no está especificada: ¿el agrupamiento ocurre antes o después del chunking?
Fix: Especificar el pipeline agrupamiento-luego-chunking explícitamente. Definir conteo mínimo de caracteres por grupo. Declarar estos como defaults a afinar post-MVP.
Medium

Soft-delete crea citas desactualizadas

§4.1 step 7, §4.6
Con la política de borrado soft por defecto, los embeddings de mensajes borrados permanecen consultables en pgvector. El PRD no especifica si la cadena de recuperación filtra embeddings soft-deleted de los resultados de búsqueda.
Fix: Requerir explícitamente que la query de búsqueda/recuperación siempre filtre WHERE deleted_at IS NULL en el join con discord_messages.
Medium

OAuth2 redirect URI no especificado — fricción de despliegue

§4.7 Autenticación
Discord OAuth2 requiere whitelisting del redirect URI. Cada operador self-hosted corre en un dominio/IP/puerto diferente. El PRD nunca aborda cómo los operadores configuran esto, si el desarrollo localhost funciona out-of-the-box, o si la callback URL es configurable vía .env. Es el punto de fricción más común en la primera hora de instalación.
Fix: Añadir DISCORD_REDIRECT_URI a .env.example y documentar el paso de configuración del dashboard de aplicación Discord explícitamente en la guía de despliegue.
Medium

Redis en producción podría evictar eventos de indexación silenciosamente

§7.2, §4.5
El límite de memoria de Redis en prod (256M) significa que Redis podría comenzar a evictar entradas de stream. El PRD no especifica la política de evicción Redis ni qué ocurre con eventos descartados del stream.
Fix: Establecer maxmemory-policy noeviction para el Redis de cola de eventos y documentar monitoreo de stream lag. Definir proyecciones de uso de disco para las configuraciones recomendadas.
Low

docker-compose.yml expone PostgreSQL en 0.0.0.0:5432 en dev

§7.1
El compose file de desarrollo enlaza Postgres a todas las interfaces, no a 127.0.0.1:5432. En una VM cloud o host compartido, esto expone la base de datos a redes externas.
Fix: Enlazar a 127.0.0.1:5432:5432 en el compose file de dev. Añadir nota de configuración de firewall a la guía de despliegue.
Low

Interpolación ${ENV_VAR} en YAML sin especificar el parser

§4.6
El YAML de configuración referencia "${DISCORD_GUILD_ID}". YAML no soporta natively la sustitución de variables de entorno — requiere Docker Compose interpolation, un loader custom, o una librería. El PRD no especifica cuál mecanismo maneja esto.
Fix: Especificar en §4.6 que loader.ts resuelve patrones ${VAR} contra process.env al startup, y que .env se carga antes de que corra el parser YAML.
Low

backfill_limit: 1000 tiene unidad ambigua

§4.6 config
¿1000 mensajes, 1000 páginas API (50 msgs cada una), o 1000 embeddings? Con 20 canales habilitados y 10 páginas por canal, el bot ejecuta 200 llamadas API al arrancar — potencialmente disparando el rate limit global de Discord.
Fix: Documentar la unidad explícitamente (mensajes, no páginas), especificar el delay entre requests durante backfill, y añadir recomendación para guilds grandes de habilitar backfill en horario escalonado.

Notas mecánicas

  • Drift de glosario — "fragmento" (§2.3, §4.8), "documento" (§4.3, §16), "embedding" (§4.2, §4.8, §5.1), "chunk" (§4.6 chunk_size) refieren al mismo concepto indexado en distintos niveles de abstracción. Sin término canónico establecido.
  • IDs no-continuos — SB-1 a SB-17 cruza tres componentes (Discord Bot: SB-1–5, SB-16/17; Backend: SB-6–10; Web App: SB-11–15) en un único namespace sin mapeo visible. SS-1–9 y SNF-1–17 son namespaces separados que podrían colisionar sin prefijo.
  • SNF no-continuo — SNF-6–9 (disponibilidad) aparecen en §11.2 después de SNF-1–5 (performance) en §11.1, luego SNF-10–12 (testing) en §11.3, luego SNF-13–17 vuelven a §11.1. La numeración no sigue el orden de sección.
  • Gap de cross-reference — §13 Threat Model menciona "Lista de canales a excluir" pero no existe tal lista en §4.6; el mecanismo es enabled: false por canal.
  • Duplicación SM/NFR — SM-3 = SNF-1 (búsqueda vectorial P95 <200ms); SM-4 = SNF-2 (agente P95 <5s); SM-5 = SNF-10 (cobertura >80%). Listados en dos secciones sin cross-reference.
  • docker-compose.prod.yml — DISCORD_CLIENT_SECRET aparece tanto en el bloque environment: como en secrets:. La variable de entorno tomaría precedencia sobre el montaje del secret, haciendo la declaración del secret redundante.
  • Typo — §4.2 "ENsamBLAR CONTEXTO" — casing mixto en header de sección.
