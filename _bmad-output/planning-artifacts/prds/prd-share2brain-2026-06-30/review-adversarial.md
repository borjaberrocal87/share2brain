# Adversarial Review — Share2Brain Self-Hosted

## Preamble

The PRD is architecturally coherent for a solo-developer portfolio project, but it conflates "well-specified" with "well-thought-out": the success metrics are invented numbers with no validation basis, the MVP scope is drastically larger than a true MVP, and the read-tracking feature is a ticking time-bomb at any non-trivial community size. The single biggest risk is that the team ships a Docker Compose stack nobody can realistically deploy and maintain without DevOps expertise, directly contradicting the "despliegue trivial" promise.

---

## Findings

### Critical

- **Success metrics are fabricated, not validated** (§16 Métricas de éxito) — SM-1 (">70% preguntas resueltas sin intervención humana") and SM-2 (">95% responses with cited source") are presented as targets with no prior baseline, user research, or benchmark to justify them. RAG systems on real community data routinely fail to find relevant chunks for ambiguous or jargon-heavy questions; 70% resolution without fallback is optimistic even for mature production systems. SM-7 (">60% documentos marcados como leídos") assumes users will voluntarily track their reading — a behavior with zero precedent in Discord community culture. *Fix:* Replace hard targets with instrumented proxies you can actually measure at launch (e.g., "user does not follow up with the same question within 5 minutes"), and explicitly label all SM targets as "aspirational — to be validated in beta."

- **Read tracking will generate a data explosion at community scale** (§4.8, §5.2) — `user_read_status` stores one row per (user, embedding) pair. A community with 500 active members and 50,000 embeddings (easily reached with `backfill_limit` and ongoing indexing) produces 25,000,000 rows in a single table. The `mark-all` endpoint (`POST /api/read-status/mark-all`) then needs to insert potentially 100,000 rows in a single transaction for one user marking one channel read. No batch size limit, no pagination, no upper bound is specified. The two indexes on `user_read_status` will not save a full-table scan at this cardinality. *Fix:* Define a maximum community size this feature is designed for. Consider a bitset approach (one row per user+channel with a bloom filter or bitmask) or an "unread since timestamp" inverse model instead of positive-acknowledgment per embedding.

- **MVP is not actually an MVP** (§15 Hoja de ruta) — The MVP scope listed includes: Discord bot + backfiller + sync, web search UI, chat with RAG agent, read tracking, RBAC-by-channel (wait — this is v1?), YAML configuration, Telegram/Slack notifications, Docker Compose packaging, and "basic tests." That is a multi-month engineering effort for a solo developer, not a shippable slice. More critically, the roadmap says RBAC and sync of edits/deletions are v1, but the system spec (§4.1 SB-16, SB-17, §4.3, §9 SS-9) fully specifies them as required components — creating a contradiction that will cause scope arguments on every PR. *Fix:* Define MVP as the smallest deployable artifact where value is demonstrable: bot indexes + chat with citations. Move read tracking and RBAC to v1. Make the MVP/v1 boundary explicit in every section that references it.

- **LLM API outage is unhandled** (§4.2, §11.2 Disponibilidad) — The agent runtime calls an external LLM API (Anthropic/OpenAI). If that API goes down or returns 5xx, the chat endpoint presumably errors out. The PRD specifies graceful shutdown (SNF-8) and restart on failure (SNF-7), but neither covers the scenario of a healthy pod with an unavailable upstream LLM. There is no defined circuit breaker, no degraded-mode fallback (e.g., "serve only search results, disable chat"), and no user-facing error contract. The health check at `/health` reports `"notifier": "connected"` but does not appear to test LLM API reachability. *Fix:* Define explicit failure behavior for LLM unavailability. At minimum: return a user-visible error with retry guidance, surface LLM API status in `/health`, and document the degraded-mode behavior.

### High

- **Discord Gateway outage breaks real-time indexing silently** (§4.1, §13 Threat Model) — The bot relies on the Discord Gateway WebSocket for `messageCreate` events. If the bot disconnects (Discord outage, rate-limit reconnect storm, token expiry), messages are silently missed. The PRD mentions "Sync on start" (step 8 in the indexing flow) to catch missed events on restart, but this only works if the operator notices the bot is down and restarts it. The `backfill_limit` is a single integer — there is no specification of how the Backfiller knows which messages it already indexed vs. which are new, beyond `indexed_at`. If the bot is down for 48 hours in a busy guild, the backfill gap could exceed `backfill_limit`, creating permanent blind spots. *Fix:* Specify the reconciliation algorithm precisely: store the last-seen Discord message snowflake per channel, and backfill from that point rather than from a fixed count. Define a monitoring alert when the bot has not received any event for N minutes.

- **RBAC cache staleness creates a security window** (§4.6, §9 SS-9, SD-14) — The `user_roles_cache` has a configurable TTL (default 5 minutes). If a Discord admin removes a role from a user (e.g., demoting a mod who is actively using the web app), that user retains access to restricted channels for up to 5 minutes. In a disciplinary scenario this is a real problem. The threat model lists "usuario accede a contenido de canales restringidos" as mitigated, but the mitigation is incomplete — it defers to a configurable TTL without specifying the minimum enforceable TTL or providing a manual invalidation mechanism. *Fix:* Add a cache-invalidation endpoint (admin-only) to force role refresh for a given user. Document the security trade-off explicitly: "TTL below 60s will cause Discord API rate limit pressure."

- **Discord API rate limits for `guilds/{guild_id}/member` are not quantified** (§4.7 Autenticación) — Every login triggers a call to `GET /users/@me/guilds/{guild_id}/member` to fetch roles. Discord's rate limit for this endpoint is 1 request per second per route, with burst limits. In a community with 100 users logging in after a service restart, this will queue-up or fail. The PRD acknowledges rate limits for backfill (SNF-5) but not for the auth flow. *Fix:* Implement a per-login debounce: if a cached role entry exists and is not yet expired, skip the Discord API call on login. Document the rate limit ceiling and the expected login throughput the system can handle.

- **No backup or recovery strategy is specified** (§12 Despliegue, §17 Checklist) — The deployment checklist mentions "Backup de base de datos programado" as a post-deploy checkbox, but the PRD contains zero specification of: backup frequency, retention policy, restore procedure, or what happens to `user_read_status` data after a restore (will read state be inconsistent with the actual embeddings?). For a self-hosted product, the operator is responsible for this, yet the PRD provides no guidance. A corrupt or lost Postgres volume means re-running backfill and losing all read tracking history. *Fix:* Add a section defining the minimum backup contract: daily `pg_dump`, volume snapshot cadence, and a documented restore procedure. Make backup configuration part of `Share2Brain.config.yml`.

- **Conversation history "summary memory" is unspecified** (§4.2 Agent Runtime) — The agent assembles context using "Historial de conversación (summary memory)" but the PRD never defines: what summarization strategy is used, how many prior turns are kept before summarization kicks in, what the maximum context window budget is, or how the summary is stored. A summary that loses key earlier context can cause the agent to contradict itself mid-conversation. This is the mechanism most likely to produce degraded UX, yet it has zero specification. *Fix:* Specify the memory strategy concretely: rolling window of last N messages, or LangChain's ConversationSummaryBufferMemory with explicit token budgets. Tie it to the LLM's context window limit for the configured model.

- **`search_web` tool in MVP with `permission: auto` is a liability** (§4.6 config example) — The example config enables `search_web` as a tool with permission `"auto"`, meaning the agent can invoke web search without user confirmation. This tool is not in the "No-objetivos" list, yet it introduces external dependencies (a web search API not named in the stack), unpredictable cost, potential for the agent to cite external sources that contradict the community's own knowledge, and a new API key management burden. The PRD never specifies which web search provider backs this tool. *Fix:* Move `search_web` to the "Posterior" roadmap phase. If it stays, name the provider, define cost controls, and set default permission to `"ask"` not `"auto"`.

### Medium

- **Chunking strategy at 500 tokens is arbitrary** (§4.6 `knowledge.chunk_size: 500`) — Discord messages range from 1 character to 2000 characters. A 500-token chunk with 50-token overlap is a LangChain default, not a Discord-optimized choice. Short conversational messages (the majority of Discord content) will be padded or over-fragmented; long technical answers will be split at arbitrary boundaries. The `grouping_window: 10` parameter (group messages in 10-message windows) is promising but its interaction with `chunk_size` is unspecified — does grouping happen before or after chunking? *Fix:* Specify the grouping-then-chunking pipeline precisely. Define a minimum message-group character count to avoid creating embeddings from near-empty chunks. Reference a validation experiment or at least state these are defaults to tune post-MVP.

- **Soft-delete creates stale citations** (§4.1 step 7, §4.6 `delete_policy: "soft"`) — With the default `soft` delete policy, embeddings from deleted messages remain queryable in pgvector. The RAG agent can cite a message that no longer exists in Discord, presenting a broken link or misrepresenting removed content. The PRD does not specify whether the retrieval chain filters out soft-deleted embeddings from search results. *Fix:* Require the search/retrieval query to always filter `WHERE deleted_at IS NULL` on joined `discord_messages`. Add this as an explicit requirement, not an assumed implementation detail.

- **No session expiry enforcement is specified** (§5.2 sessions table) — The `sessions` table has an `expires_at` column, but the PRD never specifies: how long sessions last (1 day? 30 days?), whether there is a cron job to purge expired sessions, or what happens to an API call with an expired session token. The `idx_sessions_expires` index exists, suggesting periodic cleanup was intended, but the cleanup job is absent from the repo structure and the checklist. *Fix:* Specify session TTL as a configurable value. Define the cleanup mechanism (e.g., a scheduled job or on-access lazy expiry) and include it in the checklist.

- **OAuth2 redirect URI is not specified, creating deployment friction** (§4.7 Autenticación) — Discord OAuth2 requires the redirect URI to be whitelisted in the application dashboard. For self-hosted deployments, every operator runs on a different domain/IP/port. The PRD never addresses how operators configure this, whether localhost development works out of the box, or whether the callback URL is configurable via `.env`. This is a common first-hour friction point that will block new operators immediately. *Fix:* Add `DISCORD_REDIRECT_URI` to `.env.example` and document the Discord application dashboard configuration step explicitly in the deployment guide.

- **No disk-full handling** (§12.2 Requisitos del servidor) — The minimum disk recommendation is 20 GB SSD. A community running for 12 months with active backfill, conversation history, and Postgres WAL can easily exhaust this. Redis persistence (`redis_data` volume) is enabled but its memory limit (256M in prod) means Redis will start evicting stream entries — silently dropping indexing events. The PRD does not specify Redis eviction policy or what happens to events dropped from the stream. *Fix:* Set `maxmemory-policy noeviction` for the event-queue Redis and document monitoring for stream lag. Define disk usage projections for the recommended configurations.

- **The Notifier service has no failure isolation** (§4.5, §10.3) — The Notifier (Telegram/Slack) is shown as a sibling service in the architecture diagram but its failure behavior is unspecified. If Telegram is unavailable, does it block the `knowledge.events` consumer, causing Redis Stream consumer lag to grow? The PRD states notifications are optional to configure but does not clarify whether the Notifier consumer is always running and what happens when configured destinations are unreachable. *Fix:* Specify that notification failures must not block event processing. The Notifier should be fire-and-forget with dead-letter handling, not a synchronous dependency.

- **`mark-all` endpoint has no authorization scope defined** (§4.2 Endpoints) — `POST /api/read-status/mark-all` is listed as marking a channel as read, but the endpoint contract does not specify: does the user need to have RBAC access to the channel to mark it read? Can a user call this endpoint for a channel they cannot see in search results? The symmetry between read-access and mark-as-read is assumed but not specified. *Fix:* Add explicit authorization rules to each read-status endpoint: a user can only set or clear read status for embeddings in channels they are permitted to access under RBAC.

### Low

- **docker-compose.yml exposes PostgreSQL on 0.0.0.0:5432 in dev** (§7.1) — The development compose file binds Postgres to all interfaces (`"5432:5432"`), not `127.0.0.1:5432`. On a cloud VM or shared host, this exposes the database to external networks. Operators told to "just run docker compose up -d" without a firewall will have an open Postgres port. *Fix:* Bind to `127.0.0.1:5432:5432` in the dev compose file. Add a firewall configuration note to the deployment guide.

- **Config file uses `${ENV_VAR}` interpolation without specifying the parser** (§4.6 `Share2Brain.config.yml`) — The YAML config references `"${DISCORD_GUILD_ID}"` syntax. YAML does not natively support environment variable substitution — this requires either Docker Compose variable interpolation (only works when loaded by Compose, not by the app directly), a custom loader, or a library like `dotenv-expand`. The PRD does not specify which mechanism handles this, leaving implementers to discover the contract at build time. *Fix:* Specify explicitly in §4.6 that `loader.ts` resolves `${VAR}` patterns against `process.env` at startup, and that `.env` is loaded before the YAML parser runs.

- **No versioning strategy for `Share2Brain.config.yml`** (§4.6, SD-2) — As the product evolves, the YAML schema will change. Operators who upgrade the Docker image but do not update their config file will get silent failures or unexpected behavior. The PRD specifies Zod validation but not what happens when a config field is removed or renamed between releases. *Fix:* Add a `version: "1.0"` field to the config schema and define a migration strategy (warn on deprecated fields, error on removed required fields).

- **Attachments and embeds are stored as JSONB but never indexed** (§5.2 `discord_messages`) — The `attachments` and `embeds` columns capture rich content (images, link previews, file attachments), but the indexing pipeline (§4.1 step 3) only creates embeddings from `chunk_text`, which derives from message `content`. An image attachment with no alt text or description contributes zero knowledge to the RAG index. This is an implicit limitation that operators will not anticipate. *Fix:* Add a non-goal statement: "attachment content (images, files) is not indexed in v1 — only text content of messages is searchable."

- **`backfill_limit: 1000` is per-channel, per-run, but the unit is ambiguous** (§4.6 config) — Is 1000 the number of messages, the number of API pages (50 messages each), or the number of embeddings? Discord's channel history API returns messages in pages of up to 100. If `limit: 1000` means 1000 messages and the bot fetches 10 pages per channel, a guild with 20 enabled channels runs 200 API calls on startup — potentially triggering Discord's global rate limit (50 requests/second). *Fix:* Document the unit explicitly (messages, not pages), specify the inter-request delay used during backfill, and add a recommendation for large guilds to enable backfill on a staggered schedule.
