- (decision) SD-1: Discord como fuente de conocimiento — no requiere ingestión manual; actualización automática
- (decision) SD-2: Config via YAML — Infrastructure-as-Code; Git-friendly; para público técnico
- (decision) SD-3: Web para búsqueda y chat — interfaz accesible; no requiere Discord para buscar
- (decision) SD-4: Telegram para notificaciones — simple; gratuito; API HTTP directa
- (decision) SD-5: Single-tenant — un despliegue = una comunidad; simplifica todo
- (decision) SD-6: LangChain.js — framework estándar; tools, retrieval, memory pre-construidos
- (decision) SD-7: Read tracking por usuario — cada miembro tiene su propio progreso de lectura
- (decision) SD-8: Docker Compose — despliegue trivial con un comando
- (decision) SD-9: pgvector para embeddings — nativo de PostgreSQL; sin servicio externo
- (decision) SD-10: Sync de ediciones/borrados — fiabilidad de citations; contenido obsoleto rompe SO-3
- (decision) SD-11: delete_policy configurable — soft para historial, hard para limpieza
- (decision) SD-12: Sync on start — cobertura de eventos perdidos durante downtime
- (decision) SD-13: RBAC por canal en config — control explícito; el operador define quién ve qué
- (decision) SD-14: Cache de roles con TTL — balance entre precisión y performance
- (decision) SD-15: default_policy: deny — seguridad por defecto; denegar si no hay regla explícita

- (decision) SD-17: MVP cubre todo el alcance especificado en el PRD actual incluyendo sync, RBAC y read tracking — §15 actualizado en consecuencia
- (decision) SD-18: SO-7 'Datos bajo control' acotado a datos en reposo; procesamiento LLM delegado al proveedor configurado por el operador — no es fuga de datos sino delegación explícita
